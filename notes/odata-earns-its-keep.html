When OData actually earns its keep

A defense of unfashionable tech. Where OData shines on internal enterprise APIs, and where I still reach for plain REST.

Feb 2026 · 5 min read · Nejat Berk Şener

OData is one of those technologies that developers of my generation tend to have strong opinions about before they've seriously used it. Mostly the opinion is "isn't that a Microsoft thing from 2010?" Yes. And also — for a narrow but real set of problems — it's still the right tool.

I've been writing OData endpoints for the better part of two years on internal enterprise APIs. What follows is an honest account of where it paid off, where it didn't, and how I decide between OData and plain REST today.

The case OData was built for

Here's the setup OData handles better than anything else I've tried:

  • You have a large, tabular dataset (employees, transactions, assets, training records).
  • Five different internal consumers want five different slices of it — this filter, that sort, these columns, paginated like so.
  • The consumers are not external and will not try to DoS you with $expand=users/trainings/modules.
  • You do not want to ship a new endpoint every time someone in Finance wants a new pivot.

OData gives you $filter, $select, $orderby, $top, $skip, and $expand for free. On a well-modeled entity set, you write the controller once and every reporting consumer self-serves their query from the URL. In .NET Core the LINQ provider translates most of it straight into SQL. It feels a bit like giving your clients read-only SQL without actually giving them read-only SQL.

Where it pays off in practice

Three concrete wins I can point to:

Reporting dashboards stop being an eng ticket. Every time the business asks "can we also break it down by region," the answer is "change your URL" instead of "two weeks and a deploy." That alone has bought me back more hours than any single other technology choice.

Integrations get smaller. When an internal consumer only needs three fields and only for active users created this quarter, they only fetch those three fields for those users. No more 2MB JSON payloads where somebody .map(u => u.name)s and drops the rest.

It composes with your auth model. You can layer row-level security, company-scoping, or audit filters before the OData query runs and it Just Works. The filter tree the client sent is still applied on top.

Where I still reach for plain REST

I'd never put OData on a public-facing API. The reasons are the usual ones plus one specific to enterprise work:

The query surface is huge and harder to reason about. A generic $filter with and, or, contains(), nested predicates — this is essentially a query language, and query languages are attack surface. For a public API you almost always want a fixed, small set of named parameters with known bounds.

Performance budgets are harder to set. "GET /users" is one query. "GET /users?$filter=orders/any(o: o/total gt 1000)&$expand=orders,addresses" is a completely different query, and you won't know which one a client ran until it already ran. For internal consumers on a known-shape database, that's manageable. For unknown consumers, it's a performance pager waiting to go off.

Documentation is always a little off. OpenAPI + OData is a tenuous relationship. You can generate docs but the expressive query language is never quite what the spec says. For an internal team who reads the source, this is fine. For an external developer at 11pm, it isn't.

My current rule of thumb

For a new internal API, I now ask three questions before deciding:

Is this fundamentally a tabular read over a known entity, or is it an action / workflow? If it's tabular, OData is plausible. If it's an action (create order, approve request, trigger job), just use REST or RPC.

Is the consumer trusted and operationally close? If a backend team at a sister company is going to integrate against it next year, OData is great. If a mobile app is going to ship it to phones, probably not.

Would I otherwise end up writing 15 GET /users/by-department-and-role-and-… endpoints? If yes, I've already lost to OData. Might as well adopt it intentionally.

Not every hype cycle lands on the right technology, and not every unfashionable technology deserves to be forgotten. OData, for the narrow case it was designed for, is still quietly one of the most productive tools I have. Disagree? I'd genuinely like to hear it.
